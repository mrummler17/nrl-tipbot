## Architecture

NRL TipBot is a static GitHub Pages app generated by `generate.js`.

- `data/briefing.json`: durable round content, charity ledger, panels, tips, and narrative.
- `data/live-round.json`: current round tracker, sources, fixture board, alerts, and notes.
- `generate.js`: Node static generator that merges the data files and writes `dist/index.html` plus `dist/README.md`.
- `dist/index.html`: generated dashboard output.
- `index.html`: GitHub Pages entry file; must be copied from `dist/index.html` before deploy.
- `README.md`: normal project usage and weekly update workflow.
- `HANDOVER.md`: active continuity handover.
- `DISASTER_RECOVERY.md`: recovery procedure and resilience notes.
- `scripts/backup-recovery-pack.sh`: local backup pack creator.

There is no server process, database, queue, API backend, auth system, or runtime scheduler in the repo.

## Restore From Total Machine Loss

1. Clone the repo:

   ```bash
   git clone https://github.com/mrummler17/nrl-tipbot.git
   cd nrl-tipbot
   ```

2. Confirm branch and remote:

   ```bash
   git status --short --branch
   git remote -v
   git log --oneline -5
   ```

3. Validate JSON:

   ```bash
   node -e 'JSON.parse(require("fs").readFileSync("data/briefing.json","utf8")); JSON.parse(require("fs").readFileSync("data/live-round.json","utf8")); console.log("json ok")'
   ```

4. Build:

   ```bash
   npm run build
   ```

5. Sync the Pages entry file:

   ```bash
   cp dist/index.html index.html
   ```

6. Optional local preview:

   ```bash
   python3 -m http.server 4173
   ```

   Open `http://127.0.0.1:4173/index.html`.

7. Deploy by committing and pushing `main`:

   ```bash
   git add data/briefing.json data/live-round.json dist/README.md dist/index.html index.html README.md HANDOVER.md DISASTER_RECOVERY.md
   git commit -m "Update TipBot"
   git push origin main
   ```

8. Verify GitHub Pages:

   ```bash
   gh api repos/mrummler17/nrl-tipbot/pages/builds --jq '.[0] | {status: .status, error: .error.message, commit: .commit, created_at: .created_at, updated_at: .updated_at}'
   curl -L https://mrummler17.github.io/nrl-tipbot/
   ```

## Backup Procedure

Run a local recovery backup:

```bash
./scripts/backup-recovery-pack.sh
```

Optional custom output directory:

```bash
NRL_TIPBOT_BACKUP_DIR=/Volumes/ExternalBackups/nrl-tipbot ./scripts/backup-recovery-pack.sh
```

The backup pack contains:

- Source files needed to rebuild the static app.
- `data/*.json` current runtime content.
- Generated `dist/` and root `index.html`.
- `README.md`, `HANDOVER.md`, `DISASTER_RECOVERY.md`, `.env.example`.
- Metadata snapshots for git status, remotes, latest commit, tool versions, env-file names detected, log path inventory, and launchd/system-job notes.

The backup pack intentionally excludes:

- `.git/`
- `node_modules/`
- `.env` and `.env.*`
- `backups/`
- `*.tar.gz`, `*.zip`
- log file contents by default

Treat backup archives as sensitive because they contain operational project state and potentially future runtime data.

## Critical Logic Not To Refactor Casually

- Do not move the Mark Hughes Foundation charity ledger without recorded odds, stake, result, and explicit user confirmation before kickoff.
- Do not count external slips or unplaced leans as charity bets.
- Keep `index.html` synced from `dist/index.html` before deploy; GitHub Pages serves root `index.html`.
- Keep `data/briefing.json` and `data/live-round.json` as the durable content source.
- Re-check current NRL sources before publishing any new round, late-mail update, or result wrap.
- Do not publish historical final tip counts unless results have been cleanly cross-checked.

## Known Risks

- Static app depends on manual weekly source updates.
- Sports/team-list information changes quickly, especially late mail and Origin back-up rounds.
- GitHub Pages production can lag briefly after push.
- No automated CI currently validates JSON/build on pull or push.
- No automated scheduled backup currently runs.
- No machine-independent secret store is needed now, but GitHub credentials are still required for deploy.

## Recommended Resilience Next Steps

- Add a GitHub Actions workflow that runs JSON parse and `npm run build` on push.
- Add a scheduled reminder or automation for weekly source refresh and result wrap.
- Store periodic backup archives outside the repo, for example external drive or encrypted cloud storage.
- Consider a small `restore-check.sh` script if this app grows beyond static generation.
